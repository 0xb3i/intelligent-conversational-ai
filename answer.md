# HW3 思考题 3 解答

## 1. 百万级上下文出现后，是否可以把整个知识库直接塞进 Prompt，而不再需要 RAG？

不能简单得出“可以完全替代 RAG”的结论。超长上下文确实扩大了 Prompt 可承载的信息量，但在真实系统里，RAG 仍然有明显价值。

### （1）成本角度
- 即使模型支持百万级上下文，输入这么多 token 的成本仍然很高。
- 如果每次问答都把整个知识库送进去，费用会随着调用次数线性增长，远高于“先检索、再只喂相关片段”的做法。
- 对企业系统来说，这种方案通常难以长期承受。

### （2）时延角度
- 超长上下文会显著增加推理延迟。
- 用户提一个很小的问题，却要让模型通读整个知识库，吞吐量和响应速度都会变差。
- 在线问答、客服、搜索增强系统通常更关心首 token 时间和总体响应时间，因此仍倾向于“检索后压缩输入”。

### （3）注意力有效性角度
- “能放进去”不等于“能有效利用”。
- 上下文越长，模型越可能受到信息干扰，出现“找不到关键证据”“注意力分散”“被无关内容稀释”的问题。
- 尤其当知识库包含大量相似段落时，模型未必比专门的检索器更擅长先定位最相关证据。

### （4）可更新性与工程性角度
- RAG 的知识与模型参数解耦，文档更新后重建索引即可，系统可以快速生效。
- 如果每次都塞全库，虽然也能看到最新文档，但工程上不利于缓存、分层召回、权限控制和日志分析。
- RAG 更容易做“召回了哪些文档、为什么答成这样”的可观测性建设。

### （5）权限与隔离角度
- 很多业务场景要求不同用户只能访问不同文档子集。
- 把整个知识库全部放进 Prompt，天然不利于细粒度权限控制，也增加敏感信息误暴露风险。
- RAG 可以先做过滤，再做召回，更适合企业知识库。

### （6）什么场景下可以少用 RAG？
- 知识库本身很小，比如几十篇短文档。
- 查询任务偏总结、浏览，而不是精准定位。
- 对成本和时延不敏感，只追求快速原型验证。

### 结论
超长上下文会削弱部分 RAG 的必要性，但很难在大多数真实生产环境中完全替代 RAG。更现实的趋势是：**长上下文 + RAG 混合使用**。长上下文负责更强的整合与推理，RAG 负责高效、低成本、可控地提供相关证据。

## 2. 将过滤字段拼到文档末尾，再用 BM25 检索，这种做法如何评价？Elasticsearch 如何做？向量检索方便吗？

### （1）这种做法的优点
把作者、年份、标签等结构化字段附加到文档末尾，确实能让 BM25“感知”这些字段，从而在某些简单场景下同时考虑关键词和属性。

比如查询里包含“2023 年 张三写的论文”，那么把“作者:张三 年份:2023”拼进去，BM25 有机会把相关文档排到更前面。

### （2）主要问题
但这种方案本质上是**拿文本匹配去模拟结构化过滤**，问题很多：

#### ① 过滤不严格
- 用户要求“只在 2023 年文档里检索”，这本质上是布尔约束。
- 但 BM25 只能给相关性分数，不能保证结果严格满足过滤条件。
- 可能某篇 2022 年文档因为正文更匹配，最终仍排到前面。

#### ② 字段语义被污染
- 把结构化字段直接拼到正文，会改变文档词项分布。
- 原本只想让“年份/作者”用于过滤，结果它们还参与了正文相关性打分，可能扰动排序。

#### ③ 维护性差
- 一旦字段变多、值很长、过滤条件复杂（范围查询、多值枚举、布尔组合），拼接法会迅速失控。
- 比如“年份在 2021~2023 之间且作者不是张三”，就很难优雅表达。

#### ④ 无法利用专门索引结构
- 结构化字段本来适合倒排、列存、数值范围索引等专门结构。
- 强行转成文本，会损失这些结构带来的效率和准确性。

### （3）Elasticsearch 常见做法
Elasticsearch 的标准思路是：

- **文本字段**走倒排索引，用于全文检索和相关性排序；
- **结构化字段**单独建 field，用于 `filter` / `term` / `range` / `bool` 查询；
- 通常是“**先过滤，再在候选集合上做检索/排序**”或者“检索与过滤共同组成查询计划”。

例如：
- `must`: 正文匹配某些关键词
- `filter`: 年份 = 2023，作者 = 张三

这样做的优点是：
- 过滤条件严格满足
- 过滤部分通常不参与打分，可缓存，效率高
- 文本相关性和结构化约束职责清晰

### （4）向量检索方便处理这种问题吗？
相对来说，**不如传统倒排检索天然方便**。

原因是：
- 纯向量相似度只擅长“语义近邻”，不擅长精确布尔过滤。
- HNSW 这类 ANN 索引的核心目标是近似最近邻搜索，不是复杂条件筛选。
- 如果要加过滤，常见做法通常是：
  1. 先基于元数据过滤；
  2. 再在过滤后的子集里做向量检索；
  3. 或者做混合检索与后重排。

但当过滤后的候选集合变化很频繁、组合很多时，ANN 的效率和实现复杂度都会上升。

### 结论
把结构化信息拼到文档末尾，只能算一种临时性的“取巧方案”，适合小实验，不适合严肃系统。工业界更主流的做法是：**结构化字段独立索引，文本字段独立检索，二者在查询层做组合**。对于这类“先过滤后检索”问题，传统倒排系统通常比纯向量检索更成熟、更自然。

## 3. 为什么 Coding 工具在代码检索上，从早期依赖向量检索，转向更多依赖 find、grep 等传统命令？

### （1）代码检索的目标和自然语言检索不同
通用向量检索更擅长“语义相似”，但代码检索很多时候需要的是：
- 精确符号定位
- 文件名匹配
- 类/函数/变量名查找
- import 关系定位
- 错误字符串、日志、配置项搜索

这些任务本质上对**字面匹配和结构定位**要求很高，grep/find 往往比向量检索更直接。

### （2）代码中关键词本身信息量很强
在自然语言里，同义改写很多，语义检索优势明显；但在代码里：
- 函数名
- 类名
- 路由名
- 配置键
- 错误码
- SQL 字段名

这些 token 通常非常关键，而且往往要求“一个字符都不能错”。传统文本检索在这方面天然更强。

### （3）代码库可验证、可遍历
代码仓库不是开放语料，而是结构清晰的文件系统：
- 可以按目录缩小范围
- 可以按扩展名过滤
- 可以按 import / symbol / file path 继续追踪

因此 agent 可以通过 `find`、`grep`、`glob`、语言服务等方式逐步缩小范围，成本低、解释性强。

### （4）向量检索容易召回“看起来相关但其实没用”的代码
代码中的局部语义高度模板化，很多函数写法相似。
- 向量检索可能召回风格相似、功能相近的片段；
- 但用户真正需要的，往往是某个确切实现、某个具体调用链。

对编码场景来说，**低幻觉的精确定位**往往比“语义上差不多”更重要。

### （5）索引更新和工程维护成本
代码库变化频繁，分支切换、临时文件、未提交修改都很常见。
- 向量索引需要额外构建和同步；
- 很容易和当前工作区状态不一致；
- grep/find 直接针对当前文件系统，所见即所得。

### （6）可解释性更强
当 agent 使用 grep 找到 `fooBar` 出现在哪几个文件时，开发者很容易理解它为什么这么改。
而向量召回往往难以解释“为什么是这几个片段”，可控性较弱。

### 结论
Coding 场景逐渐边缘化向量检索，不是因为向量检索完全没用，而是因为**代码检索更强调精确、可验证、低延迟、与当前工作区一致**。因此在大多数日常编码任务里，传统工具链更符合需求；向量检索更适合作为补充，而不是主干。

## 4. 新闻资讯平台高频动态更新时，向量索引更倾向 IVF 还是 HNSW？为什么？

如果新闻数据处于“每分钟都有大量增删改”的高频更新状态，我会更倾向于 **IVF**，或者更准确地说，倾向于**更容易批量重建和维护的分层索引方案**，而不是重度依赖 HNSW。

### （1）HNSW 的优点
HNSW 的检索效果通常很好：
- 召回率高
- 延迟低
- 在静态或低频更新场景表现优秀

但它更适合“建好以后长期查询”的情形。

### （2）高频动态更新下 HNSW 的问题
#### ① 删除和更新不够友好
- HNSW 是图结构，节点之间存在邻接关系。
- 插入尚可，但频繁删除、修改会导致图结构逐渐退化。
- 很多系统对删除采用软删除或定期重建，维护成本较高。

#### ② 持续高频写入下工程复杂
- 新闻平台每分钟有大量新内容进入，旧内容下线、修订。
- 如果每次都维护高质量图连接，系统实现和资源管理会更麻烦。

### （3）为什么更倾向 IVF
#### ① 更适合批量更新与分片管理
- IVF 本质是“先聚类分桶，再桶内搜索”。
- 在高频更新系统里，更容易做增量写入、冷热分层、定时重建。
- 可以把“最新新闻”放在增量索引，把历史新闻放在稳定主索引中。

#### ② 更容易做工程折中
- 新闻场景对“绝对最优近邻”往往没那么极致，更看重可维护性、吞吐和更新能力。
- IVF 虽然检索质量可能略逊于 HNSW，但更容易和实时索引、批处理重建结合。

#### ③ 更适合多级检索架构
- 可以先通过时间、频道、语言等过滤缩小范围；
- 再用 IVF 做粗召回；
- 最后用 reranker 精排。

在新闻系统里，这种整体架构的收益通常比单独追求 ANN 指标更重要。

### （4）补充：现实里未必是“二选一”
工业界常常不是纯粹选 IVF 或 HNSW，而是：
- 最新数据：小型实时索引
- 历史数据：大规模主索引
- 定期 merge / rebuild
- 配合元数据过滤与重排

### 结论
在“极其高频动态更新”的新闻场景下，我更倾向 IVF，因为它在**删除、批量更新、分层管理和工程维护**上通常更友好。HNSW 更强，但更适合相对稳定的数据集，而不是剧烈变化的实时新闻流。

## 5. Agentic RAG 通过多轮传统向量检索，能否完全替代 GraphRAG？

不能说完全替代。Agentic RAG 很强，但它和 GraphRAG 解决的问题侧重点并不完全一样。

### （1）Agentic RAG 的优势
Agentic RAG 可以：
- 自动拆解复杂问题
- 多轮改写 query
- 分步检索不同证据
- 汇总多个片段后作答

这让它在多跳问题上比“单次检索+生成”强很多。

### （2）为什么它不能完全替代 GraphRAG
#### ① 检索对象不同
- Agentic RAG 的基础仍然通常是“文本块相似性检索”；
- GraphRAG 额外显式建模了实体、关系、社区结构。

如果问题本身依赖“谁和谁有关系、经过几跳连接、路径是什么”，图结构会更自然。

#### ② 多跳推理的稳定性不同
Agentic RAG 依赖 agent 自己决定下一步查什么，这种过程：
- 灵活，但不稳定；
- 容易因为第一步召回偏了，后面越走越偏；
- 也更消耗 token 和时间。

GraphRAG 则把一部分结构先离线建好，能在查询时直接沿关系图找证据，路径更明确。

#### ③ 全局关系整合能力不同
GraphRAG 的强项之一是：
- 发现跨文档实体关联
- 聚合社区级别摘要
- 支持“主题—实体—关系”多层级视角

如果只靠 Agentic RAG 多轮检索，很多全局关系需要临时在上下文中拼出来，效率和稳定性都可能不如显式图谱。

### （3）什么时候 Agentic RAG 已经够用了
- 数据规模不大
- 实体关系不复杂
- 复杂问题占比不高
- 更在意系统简单性，而不是最强的多跳能力

这时，用 Agentic RAG 可能已经性价比很高，没必要额外做图谱。

### （4）什么时候 GraphRAG 更有必要
- 查询经常涉及多实体、多文档、跨主题关系
- 希望回答时给出更清晰的关系路径和证据链
- 知识库比较稳定，值得做离线结构化建设
- 需要社区发现、全局摘要、关系导航

### 结论
Agentic RAG 可以在很多场景下**部分替代** GraphRAG，尤其适合快速搭建和中等复杂度任务；但它不能完全替代 GraphRAG。两者更像是：**Agentic RAG 用动态策略弥补检索不足，GraphRAG 用显式结构降低多跳推理难度**。在复杂关系密集型知识库中，图谱仍然有独特价值。

## 6. 多模态 RAG 中，如何提升表格、图表等非自然图像的检索质量？

CLIP 这类模型主要在“自然图像—文本对”上训练，因此对照片、场景、物体对齐较好，但对表格、流程图、统计图、UI 截图等非自然图像往往不够理想。要提升这类内容的检索效果，可以从以下几个方面改进。

### （1）不要只依赖统一视觉 embedding
对于非自然图像，单纯把图片编码成一个向量往往不够。更好的方式是把它拆解成多种可检索信息：
- OCR 文本
- 表格结构
- 图表标题、坐标轴、图例
- 页面布局区域
- 图中关键数字与实体

也就是说，要把“视觉对象”转化成“结构化语义对象”。

### （2）先做结构化解析，再检索
#### 表格
- 使用表格识别模型提取行列结构、表头、单元格内容；
- 把表格转换成 markdown / HTML / JSON；
- 同时保留原图和结构化文本双通路索引。

#### 图表
- 提取标题、轴标签、图例、数据系列、关键趋势；
- 将图表转成文本摘要，例如“2023 年销售额在 Q4 达到峰值”。

这样做后，检索时不再只靠图像向量，而是能用更适合语言检索和推理的中间表示。

### （3）采用混合检索
对于非自然图像，通常应采用：
- 视觉向量检索
- OCR 文本检索
- 结构化字段过滤
- reranker 重排

例如用户问“显示 2024 年各地区收入对比的柱状图”，系统既可以：
- 用 OCR / 文本匹配“2024、地区、收入”；
- 也可以用图表类型分类器识别“柱状图”；
- 最后综合排序。

### （4）使用领域专用模型或微调
如果任务里非自然图像占比很高，可以考虑：
- 使用文档理解模型、表格模型、图表理解模型；
- 在业务数据上继续对齐图像—文本表示；
- 让模型学会“图表/表格与说明文字”的匹配，而不只是自然照片说明。

这通常比直接拿通用 CLIP 硬做效果更好。

### （5）建立多粒度索引
非自然图像常常不是整张图都有用，而是局部区域重要。可以：
- 对整页建索引；
- 对局部区域（表格块、图例块、标题块）建子索引；
- 查询时做 coarse-to-fine 检索。

这样能提高命中精度，避免整图 embedding 把关键信息稀释掉。

### （6）生成可解释中间描述
可以让多模态模型先为表格/图表生成“机器可读 caption”，例如：
- 图表类型
- 主要趋势
- 极值
- 对比关系
- 异常点

然后把这些 caption 与原图共同存入索引。这样后续查询更容易命中真正语义。

### 结论
提升非自然图像检索质量的关键，不是继续强行依赖 CLIP 式统一 embedding，而是**把图像中的文字、结构、布局和数据语义显式提取出来，构建混合表示与混合检索链路**。对于表格、图表这类内容，“文档理解 + 结构化抽取 + 混合召回 + 重排”通常比纯视觉向量检索更有效。
